微信公众号客服

2017-10-23

文档简介

本文档描述微信客服的界面及服务流程，预期读者为开发人员及运营维护人员。
涉及到的UI组件可在WeUI的Github页面中参考。

产品概述

微信客服为用户提供一种不通过盒子经销商，直接处理问题的方法。可以减少经销商的售后服务工作，同时建立与用户的联系渠道。

目标和意义

根据经验，盒子经销商在分销盒子时期望的点主要为：

  1. 销售利润高、退货率低
  2. 售前工作少
  3. 售后服务少

微信客服的目标是大部分问题可以让用户自主解决而无需找到经销商，以满足经销商的“售后服务少”期望；对于用户来说，微信客服使服务流程变短。
微信客服的意义在于通过满足经销商对“售后服务少”的期望，最终提高盒子销量。

用户角色

预期存在以下几类用户：

  1. 盒子用户
    盒子用户为主体用户，是服务的发起者；
    对于创朝丰 OS 的用户，可能理解能力偏低，功能和文案必须简单易懂
  2. 盒子经销商
    盒子经销商可能存在于流程中：可能承接用户服务需求，也可能直接发起服务；
    对于创朝丰 OS 的经销商，需要使用他们所习惯的语言
  3. 维护人员
    维护人员负责处理盒子用户与盒子经销商发起的服务需求

场景描述

主要应对以下场景：

  1. 用户在使用过程中发现问题，不知如何解决
  2. 用户不会使用某个功能
  3. 客户接到用户反馈，但不知道解决方法
  4. 我们解决了一个用户反馈，没有办法通知用户

需求范围

功能需求

  1. 账号与设备绑定

    • 通过二维码将微信号与设备绑定
    • 在问题上报时盒子同步上传日志

  2. 常用问题解决方法查询

    • 查看的入口和界面
    • 问题和解决方法文档
    • 数据统计

  3. 问题反馈

    • 问题反馈的入口和界面
    • 查看问题反馈，包括搜索和筛选
    • 通知问题处理状态

  4. 功能使用方法

    • 查看的入口和界面
    • 需要介绍的功能文档
    • 数据统计

非功能需求

  1. 账号申请
    以公司主体申请服务号，名称为“智能电视盒助手”

  2. 反馈问题的处理流程
    需要关注问题处理的流向，以及流程中向用户反馈处理进度的节点

    • 处理流程
      1. 用户提交问题反馈，后台记录反馈 ID、用户的联系方式、问题详情
      2. 整理问题并过滤可以通过客户支持解决的问题。（责任人：陆顾君）
      3. 对于重要且可快速解决的问题（如CCTV-1看不了），需要优先解决，解决过程根据情况通过 Bugzilla 或 Tower 跟踪；（后台只承担记录问题、与用户沟通的功能，问题的追踪由 Bugzilla 和 Tower 负责。这是为了避免范围相互交叉导致重复劳动，并且范围交叉可能导致问题无法落实到计划中。）
      4. 定期汇总一段时间内用户提交的问题，并对要解决问题进行计划，解决过程根据情况通过 Bugzilla 或 Tower 跟踪；
        不在当前版本解决的问题记录到需求池中（需要记录来源用户的联系方式）；
        不解决的问题关闭并记录，不进入需求池；
      5. 定期查看和清理需求池中不在当前版本解决的项目，决定项目处理或关闭
    • 需要反馈的节点（责任人：陆顾君）
      1. 用户提交问题时，需要立即给用户一个反馈，表示我们已经收到反馈
      2. 问题已有确定解决计划时，若超出2周，给用户一个反馈
      3. 问题已在线上版本修复时，需要给用户反馈，展示工作成果、表示对这个用户的重视，目标是让用户自发宣传产品。这些用户未来可以作为“种子用户”
    • 如何向用户反馈进度
      采用微信联系。
      可建立反馈模板，以避免不同人员的语言习惯对体验造成不一致。

设备绑定

  • 简要说明
    用户可以通过进入盒子主页-问题反馈界面扫描二维码，同时完成关注公众号和绑定设备。
    一个设备可以与多个账号绑定，但一个账号只能绑定一个账号。（这是为了保证日志的正确性。）
    （不升级的盒子的处理流程、屏蔽规则？）
    过程如图：
  • 界面内容
    略
  • 触发条件
    • 用户扫描二维码
  • 输出
    • 关注微信号
    • 设备与微信号绑定

  • 主要流程

    1. 用户点击设置-问题反馈界面
    2. 若用户Launcher版本支持发送反馈，则显示包含设备信息的二维码；
      1. 用户扫描二维码
      2. 关注微信公众号并绑定设备
    3. 若用户Launcher版本不支持发送反馈，显示之前版本的“问题反馈”，没有二维码
    4. 用户扫码后，判断账号是否已与一个设备绑定
      1. 若未绑定，用户点击“绑定设备”
        1. 与当前设备绑定
        2. 进入微信号
        3. 微信号发送消息：“您的账号已与新的设备绑定”
      2. 若已绑定，用户点击“绑定设备”
        1. 解绑之前设备，与当前设备绑定
        2. 进入微信号
        3. 微信号发送消息：“您的账号已与新的设备绑定”

  • 分支流程

    • 当账号尝试与多个设备绑定时，账号仅与最后一个设备绑定
    • 当账号与设备解绑后，重新扫描二维码，可以再次建立绑定关系

查看功能介绍

  • 简要说明
    用户可以点击公众号底栏的“功能介绍”，选择项目并进入微信文章找到功能介绍。
  • 界面内容
  • 触发条件
    • 用户点击“功能介绍”按钮后，选择一个内容
  • 输出
    • 进入相应微信文章页面
  • 主要流程
    1. 用户点击“功能介绍”
    2. 用户选择列表中的一项
    3. 跳转到相应文章界面

查看常见问题

  • 简要说明
    用户可以点击公众号底栏的“常见问题”，选择项目并进入微信文章找到常见问题的解决方法。
  • 界面内容
  • 触发条件
    • 用户点击“常见问题”按钮
  • 输出
    • 进入相应微信文章页面
  • 主要流程
    1. 用户点击“常见问题”
    2. 跳转到相应文章界面

发送问题反馈

  • 简要说明
    用户可以在微信公众号中反馈问题，若账号已绑定设备，反馈问题会触发设备上传 log。
    （目前不要求绑定设备。）
  • 界面内容 产品类型为“系统、METV直播、云视听·极光、银河奇异果、CIBN酷喵影视、其他应用”（为了避险，不仅列出系统和METV直播）。
  • 触发条件
    • 用户点击“问题反馈”按钮
  • 输出
    • 界面进入“反馈成功”页面
    • 发送问题产品、反馈内容、微信 ID
    • 后台记录收到反馈时间、生成反馈 ID
    • 在设备上触发“发送问题日志”
  • 主要流程
    1. 用户在主页点击“问题反馈”按钮
    2. 系统验证账号是否已绑定设备，若未绑定，进入“分支流程1：未绑定设备”；若已绑定，进入下一步
    3. 系统显示问题反馈页面（图4-3）
    4. 用户输入表单，其中“产品类型”为必选项、“问题描述”为必填项、“图片或视频”是选填项。
    5. 用户点击图片上传时，进入“分支流程2：上传图片或视频”，点击已选择的图片时，进入“分支流程3：预览和删除图片或视频”
    6. 用户点击“反馈”按钮
    7. 系统验证必填项是否已填写，若未全部填写，进入“分支流程4：验证内容”；若已填写，进入下一步
    8. 系统发送反馈内容（问题产品、反馈内容、微信 ID、联系方式）
    9. 若反馈失败，进入“分支流程5：反馈失败”；若反馈成功，记录收到反馈时间、记录反馈用户的微信昵称、生成反馈 ID。界面展现反馈成功页面。（图4-4）
    10. 若账号已绑定设备，触发盒子的“发送问题日志”。
  • 分支流程
    • 分支流程1：未绑定设备
      1. 用户点击“问题反馈”按钮，此时账号未绑定设备
      2. 系统显示未绑定设备确认界面（图4-5）
      3. 用户点击“继续操作”，进入主流程的第3步；点击“返回”，回到公众号主页，公众号自动推送“如何绑定设备”文章
    • 分支流程2：上传图片或视频
      1. 用户点击问题反馈页面下方的“+”
      2. 调用微信提供的拍照或从手机相册中选图接口，使用微信的相册进行选择
      3. 选择完成后，回到问题反馈页面（此时，若已选择项已满6项，则不再显示“+”按钮），进行上传。在页面下方显示已选择图片或视频，及其上传进度。
      4. 上传全部成功前，“反馈”按钮为 Disabled 状态
    • 分支流程3：预览和删除图片或视频
      1. 用户在问题反馈页面中点击已选择的图片或视频
      2. 调用微信提供的预览图片接口，使用微信相册进行预览和删除
    • 分支流程4：验证内容
      1. 用户点击“反馈”按钮，校验发现存在未填写的必填项
      2. 系统将未填写的必填项标红
      3. 系统当标红的项满足填写标准时，取消标红
      4. 用户点击“反馈”按钮，进入主流程的第6步
    • 分支流程5：反馈失败
      1. 系统显示反馈失败页面（图4-7）
      2. 用户点击“返回”按钮
      3. 系统记住用户在提交前填写的内容，回到主流程第4步

发送问题日志

  • 简要说明
    当绑定了这台设备的微信账号进行了问题反馈后，盒子发送与问题ID 绑定的 log。由于只有发生问题当时的日志有诊断价值，所以抛弃与问题反馈时间相隔太久的日志。由于用户可能不了解出错的产品类型是什么，所以发送的 log 与用户选择的产品不相关，产品类型仅用于帮助分配任务。
  • 界面内容
    上传内容为日志和非日志内容，非日志内容
  • 触发条件
    • 用户在微信客服中成功反馈一条问题
    • 盒子在线
  • 输出
    • 与反馈ID关联的日志
  • 主要流程
    1. 用户在微信公众号中点击“问题反馈”
    2. 系统通知盒子发送日志，告知反馈ID、问题产品类型
    3. 若设备在线，发送日志及关联反馈ID；若设备不在线，则不发送
    4. 系统比对收到反馈的时间及收到日志的时间，若超出允许的偏差值，则日志无效

查看反馈

  • 简要说明
    根据设想的问题处理流程，后台提供查看反馈的界面。可以通过筛选、搜索快速定位到某条反馈，对于单条反馈，可以查看反馈内容和相应的日志信息等。
    单条反馈的信息应包含：反馈ID、问题产品、问题描述、收到反馈时间、联系方式（若用户未提供联系方式，则显示微信ID）、绑定的日志
    筛选项应包含：是否包含日志、问题产品、处理状态
    搜索范围应包含：联系方式、反馈ID
  • 界面内容
  • 触发条件
    • 登录有查看权限的账号
  • 输出
    • 无
  • 主要流程
    1. 维护人员登录后台
    2. 选择“用户反馈”，展现用户反馈列表
  • 分支流程（通用的方法不再描述，仅列出）
    • 搜索：可通过“联系方式、反馈ID”进行搜索
    • 筛选：可通过“是否包含日志、问题产品、处理状态”进行筛选
    • 下载日志：若存在与反馈ID关联的日志，点击进行下载

变更处理状态

  • 简要说明
    根据设想的问题处理流程，维护人员按照问题的状态，可以对问题进行状态标记。
    （与Bugzilla打通，与tower联通的流程）
    问题需要有以下状态：

    • 未处理：问题的初始状态，代表问题未经讨论
    • 处理中：问题经讨论后，将问题标记为“处理中”状态
    • 已处理：决定处理的问题，将问题标记为“已处理”状态
    • 已解决：处理完毕的问题，在给用户发送通知后，将问题标记为“已解决”状态。
    • 已关闭：问题经讨论后决定不处理，将问题标记为“已关闭”状态

  • 界面内容

  • 触发条件
    • 维护人员点击“变更状态”按钮
  • 输出
    • 改变反馈状态
    • 触发发送通知
  • 主要流程
    1. 维护人员根据问题当前在流程中的状态，点击“变更状态”按钮
    2. 弹出选择弹窗
    3. 选择一个状态，点击保存
    4. 问题的状态变更

发送通知

  • 简要说明
    维护人员或系统能够在必要时通过微信或用户提供的联系方式通知用户问题反馈的处理状态。
    按照目前的定义，在收到反馈时及问题解决时需要给用户发送通知，如下图：
  • 界面内容
    略，通过微信后台/用户提供的联系方式进行
  • 触发条件
    • 收到反馈时
    • 反馈已处理完成时
  • 输出
    • 发送通知
  • 主要流程
    1. 后台收到用户反馈
    2. 自动发送确认通知
    3. 项目人员解决了用户反馈
    4. 维护人员查看Bugzilla或Tower中，反馈人的联系方式
    5. 若存在联系方式，优先通过用户提供的联系方式通知；若不存在联系方式，通过微信ID在微信后台-用户管理中进行通知
