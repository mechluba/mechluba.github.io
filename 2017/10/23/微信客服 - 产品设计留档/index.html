微信公众号客服 - 设计留档

2017-10-23

版本记录

时间 版本 内容简述
2017.11.1 0.2 定义微信客服的功能需求
2017.10.25 0.1 定义微信客服的需求范围

产品概述

微信客服为用户提供一种不通过盒子经销商，直接处理问题的方法。可以减少经销商的售后服务工作，同时建立与用户的联系渠道。

目标和意义

根据经验，盒子经销商在分销盒子时期望的点主要为：

  1. 销售利润高、退货率低
  2. 售前工作少
  3. 售后服务少

微信客服的目标是大部分问题可以让用户自主解决而无需找到经销商，以满足经销商的“售后服务少”期望；对于用户来说，微信客服使服务流程变短。
微信客服的意义在于通过满足经销商对“售后服务少”的期望，最终提高盒子销量。

用户角色

预期存在以下几类用户：

  1. 盒子用户
    盒子用户为主体用户，是服务的发起者；
    对于创朝丰OS的用户，可能理解能力偏低，功能和文案必须简单易懂
  2. 盒子经销商
    盒子经销商可能存在于流程中：可能承接用户服务需求，也可能直接发起服务；
    对于创朝丰OS的经销商，需要使用他们所习惯的语言
  3. 维护人员
    维护人员负责处理盒子用户与盒子经销商发起的服务需求

场景描述

主要应对以下场景：

  1. 用户在使用过程中发现问题，不知如何解决
  2. 用户不会使用某个功能
  3. 客户接到用户反馈，但不知道解决方法
  4. 我们解决了一个用户反馈，没有办法通知用户

需求范围

功能需求

  1. 账号与设备绑定

    • 通过二维码将微信号与设备绑定
    • 可以在问题上报时同步上传log
    • 可以在查看微信公众号时同步当前设备信息（将来）

  2. 常用问题解决方法查询

    • 查看的入口和界面
      提供界面上的直接入口来进行解决方法的查询，未来考虑做关键词自动回复；（关键词自动回复操作更简单，不按自动回复做的原因是列表不需要用户了解问题对应关键词。）
      解决方法详情页应用网页或微信文章形式提供，原因是存在插入图片或视频的需求
    • 问题和解决方法文档
      根据这份文档维护常用问题的列表项目和顺序。文档需要根据上线后的运营情况（包括问题的查询次数、主要上报的问题、以及来自其他渠道的问题反馈等）定期更新。
    • 数据统计
      需要统计问题浏览的频次，据此优化常用问题的项目和顺序，并对产品开发做指导。需要调研微信本身是否支持此类统计，避免重复开发。

  3. 问题反馈

    • 问题反馈的入口和界面
      提供界面上的直接入口
      反馈页面需要用网页形式进行呈现，原因是存在表单。
      界面内容上，问题的上报不宜过细（用户不了解问题产生的详细情况，且过细会使用户放弃上报），计划用户只需要填写联系方式、选择问题产品、填写问题描述。问题产品包括“盒子系统、电视直播”。
      上报问题时，触发盒子上传log，log内容由开发判断。Log需要与问题关联。
    • 后台查看问题
      后台展现问题列表，应能通过问题产品和处理状态进行筛选、通过问题ID和问题描述进行搜索，并能操作问题状态。

  4. 功能使用方法

    • 查看的入口和界面
      功能使用方法详情页应用网页或微信文章形式提供，原因是存在插入图片或视频的需求
    • 需要介绍的功能文档
      根据这份文档维护功能使用方法列表项目和顺序
    • 数据统计
      需要统计功能使用方法浏览的频次，据此优化项目和顺序。

非功能需求

  1. 触达用户的方法
    采用主页上的入口形式触达用户，以此方式可以低成本实现设备与微信号的绑定。

  2. 反馈问题的处理流程
    需要关注问题处理的流向，以及流程中向用户反馈处理进度的节点

    • 处理流程
      1. 用户提交问题反馈，后台记录反馈ID、用户的联系方式、问题详情
      2. 对于重要且可快速解决的问题（如CCTV-1看不了），需要优先解决，解决过程根据情况通过Bugzilla或Tower跟踪；（后台只承担记录问题、与用户沟通的功能，问题的追踪由Bugzilla和Tower负责。这是为了避免范围相互交叉导致重复劳动，并且范围交叉可能导致问题无法落实到计划中。）
      3. 定期汇总一段时间内用户提交的问题，并对要解决问题进行计划，解决过程根据情况通过Bugzilla或Tower跟踪；
        不在当前版本解决的问题记录到需求池中（需要记录来源用户的联系方式）；
        不解决的问题关闭并记录，不进入需求池；
      4. 定期查看和清理需求池中不在当前版本解决的项目，决定项目处理或关闭
    • 需要反馈的节点
      1. 用户提交问题时，需要立即给用户一个反馈，表示我们已经收到反馈
      2. 问题已有确定解决计划时，可以给用户一个反馈
      3. 问题已在线上版本修复时，需要给用户反馈，展示工作成果、表示对这个用户的重视，目标是让用户自发宣传产品。这些用户未来可以作为“种子用户”
    • 如何向用户反馈进度
      优先采用微信联系，目的是简化工作。对于在问题上报页面中提供联系方式的，优先通过该联系方式进行联系。
      可建立反馈模板，以避免不同人员的语言习惯对体验造成不一致。

用户故事

通过完成需求范围内的工作，用户在使用中遇到问题时的行为路径为：

  1. 用户B在使用盒子过程中出现了黑屏，通过微信号查询问题的解决方法后，解决了这个问题

    • 角色：用户
    • 场景：使用盒子过程中出现了问题
    • 行动：查看常用问题的解决方法
    • 结果：问题解决

  2. 用户C在使用盒子过程中发现一个直播频道不能播放，通过微信号没有查询到可行的解决方法，于是上报给我们，希望我们解决

    • 角色：用户
    • 场景：使用盒子过程中出现了问题，且在常用问题中没有找到解决方法
    • 行动：上报问题
    • 结果：上报问题并触发自动上传相关信息，维护人员处理问题并发布进度通知

  3. 用户A在使用投屏功能前，通过微信号查看此项功能的介绍和使用方法

    • 角色：用户
    • 场景：使用一项功能前
    • 行动：查看功能的使用方法
    • 结果：学会如何使用一项功能

设备绑定

  • 简要说明
    用户可以通过进入盒子主页-问题反馈界面扫描二维码，同时完成关注公众号和绑定设备。
  • 界面内容
  • 触发条件
    • 用户扫描二维码
  • 输出
    • 设备与微信号绑定
  • 主要流程
    1. 用户点击设置-问题反馈界面
    2. 生成包含设备信息的二维码
    3. 用户扫描二维码
    4. 关注微信公众号并绑定设备
  • 分支流程
    当账号尝试与多个设备绑定时，账号仅与最后一个设备绑定

查看功能介绍

  • 简要说明
    用户可以点击公众号底栏的“功能介绍”，选择项目并进入微信文章找到功能介绍。
  • 界面内容
  • 触发条件
    • 用户点击“功能介绍”按钮后，选择一个内容
  • 输出
    • 进入相应微信文章页面
  • 主要流程
    1. 用户点击“功能介绍”
    2. 用户选择列表中的一项
    3. 跳转到相应文章界面

查看常见问题

  • 简要说明
    用户可以点击公众号底栏的“常见问题”，选择项目并进入微信文章找到常见问题的解决方法。
  • 界面内容
  • 触发条件
    • 用户点击“常见问题”按钮后，选择一个内容
  • 输出
    • 进入相应微信文章页面
  • 主要流程
    1. 用户点击“常见问题”
    2. 用户选择列表中的一项
    3. 跳转到相应文章界面

发送问题反馈

  • 简要说明
    用户可以在微信公众号中反馈问题，若账号已绑定设备，反馈问题会触发设备上传Log。
  • 界面内容
  • 触发条件
    • 用户点击“问题反馈”按钮
  • 输出
    • 进入“反馈成功”页面
    • 记录收到反馈时间、生成反馈ID
    • 根据产品类型，在设备上触发“发送问题日志”
  • 主要流程
    1. 用户在主页点击“问题反馈”按钮
    2. 系统验证账号是否已绑定设备，若未绑定，进入“分支流程1：未绑定设备”；若已绑定，进入下一步
    3. 系统显示反馈填写页面（图4-3）
    4. 用户输入表单，其中“联系方式”为选填项、产品类型为必选项、问题描述为必填项
    5. 用户点击“反馈”按钮
    6. 系统验证必填项是否已填写，若未全部填写，进入“分支流程2：验证内容”；若已填写，进入下一步
    7. 发送反馈内容
    8. 若反馈失败，进入“分支流程3：反馈失败”
    9. 若反馈成功，记录收到反馈时间、生成反馈ID、同时触发盒子的“发送问题日志”。界面展现反馈成功页面。（图4-4）
  • 分支流程
    • 分支流程1：未绑定设备
      1. 用户点击“问题反馈”按钮，此时账号未绑定设备
      2. 系统显示未绑定确认界面（图4-5）
      3. 用户点击继续操作，进入主流程的第3步；点击返回绑定，回到公众号主页
    • 分支流程2：验证内容
      1. 用户点击“反馈”按钮，校验发现存在未填写的必填项
      2. 系统将未填写的必填项标红，并给出填写建议（产品类型：请选择产品类型；问题描述：请填写问题描述）（图4-6）
      3. 系统当标红的项满足填写标准时，取消标红
      4. 用户点击“反馈”按钮，进入主流程的第6步
    • 分支流程3：反馈失败
      1. 系统显示反馈失败界面（图4-7）
      2. 用户点击“返回”按钮
      3. 系统记住用户在提交前填写的内容，回到主流程第4步

发送问题日志

  • 简要说明
    当绑定了这台设备的微信账号进行了问题反馈后，盒子发送与问题ID绑定、与问题产品类型对应的Log。由于只有发生问题当时的日志有诊断价值，所以抛弃与问题反馈时间相隔太久的日志。
  • 界面内容
    上传内容为日志和非日志内容，非日志内容
  • 触发条件
    • 用户在微信客服中成功反馈一条问题
    • 盒子在线
  • 输出
    • 与反馈ID绑定的日志
  • 主要流程
    1. 用户在微信公众号中点击“问题反馈”
    2. 系统通知盒子发送日志，告知反馈ID、问题产品类型
    3. 若设备在线，发送对应产品类型的日志和反馈ID；若设备不在线，则不发送
    4. 系统比对收到反馈的时间及收到日志的时间，若超出允许的偏差值，则日志无效

查看反馈

  • 简要说明
    根据设想的问题处理流程，后台提供查看反馈的界面。可以通过筛选、搜索快速定位到某条反馈，对于单条反馈，可以查看反馈内容和相应的日志信息等。
    单条反馈的信息应包含：反馈ID、问题产品、问题描述、收到反馈时间、联系方式、绑定的日志
    筛选项应包含：是否包含日志、问题产品、处理状态
    搜索范围应包含：联系方式、反馈ID
  • 界面内容
  • 触发条件
    • 登录有查看权限的账号
  • 输出
    • 无
  • 主要流程
    1. 维护人员登录后台
    2. 选择“用户反馈”，展现用户反馈列表
  • 分支流程（通用的方法不再描述，仅列出）
    • 搜索：可通过“联系方式、反馈ID”进行搜索
    • 筛选：可通过“是否包含日志、问题产品、处理状态”进行筛选
    • 下载日志：若存在与反馈ID关联的日志，点击进行下载

变更处理状态

  • 简要说明
    根据设想的问题处理流程，维护人员按照问题的状态，可以对问题进行状态标记。
    问题需要有以下状态：
    • 未处理：问题的初始状态，代表问题未经讨论
    • 处理中：问题经讨论后，将问题标记为“处理中”状态
    • 已处理：决定处理的问题，将问题标记为“已处理”状态
    • 已解决：处理完毕的问题，在给用户发送通知后，将问题标记为“已解决”状态。
    • 已关闭：问题经讨论后决定不处理，将问题标记为“已关闭”状态
  • 界面内容
  • 触发条件
    • 维护人员点击“变更状态”按钮
  • 输出
    • 改变反馈状态
    • 发送通知
  • 主要流程
    1. 维护人员根据问题当前在流程中的状态，点击“变更状态”按钮
    2. 弹出选择弹窗
    3. 选择一个状态，点击保存
    4. 问题的状态变更
